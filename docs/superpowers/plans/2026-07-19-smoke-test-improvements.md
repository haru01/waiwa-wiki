## すぐ実装できるもの（判断不要）

### P7: pre-commit が案件まるごと削除を誤ってブロックする

**問題**: 今回 bank-agent を削除しようとしたところ、pre-commit の **log-append-only**（`projects/*/wiki/log.md` の既存行削除を検出）と **sources-immutable**（sources の M/D を検出）が発火し、正当な案件削除がブロックされた（`--no-verify` で回避した）。

**原因**: 両チェックとも「削除（D）」と「改変（M）」を区別していない。過去 log 行の**改変**や既存 source の**改変**は不変ルール違反だが、ファイルごと（＝案件ごと）の削除は別事象。

**対応（TDD）**:
- `Create/Modify: tools/check_immutable.py`（もしくは pre-commit にインライン）で、判定を関数化してテスト可能にする。
  - log-append-only: 対象を `--diff-filter=M`（改変のみ）に絞り、ファイル削除（D）は見逃す。
  - sources-immutable: 既存ファイルの**改変（M）**は引き続き禁止。**削除（D）**は「同一コミットで当該 `projects/<slug>/` 配下が全削除されているか」を判定し、全削除（案件廃止）なら許可、部分削除は禁止。
- `.githooks/pre-commit` を上記関数呼び出しに置換。
- テスト: ①log行改変→block ②log行追記のみ→pass ③案件全削除→pass ④source改変→block ⑤source単体削除（案件は残す）→block。

**代替（軽量）**: コード化せず「意図的な案件削除は `--no-verify`」を AGENTS.md/README に明記するだけでも可。ただし削除が日常化するなら関数化が望ましい。

### P5: derived-from の接頭辞統一と lint 追加

**問題**: `templates/hypothesis.md` の `derived-from: H-003`（接頭辞なし例）に従うと、id・wikilink・hypotheses/based-on が接頭辞つきなのに derived-from だけ短縮形になる。hwlint は derived-from を未検査のため黙って揺れる（SI-005 / self #4 と地続き）。

**対応（TDD）**:
- `Modify: templates/hypothesis.md` — `derived-from:` の例を `<PREFIX>-H-NNN`（接頭辞つき）に変更。コメントで「接頭辞つき・省略可」を明記。
- `Modify: tools/hwlint.py` — `check_frontmatter_refs` に derived-from を追加（値があれば接頭辞つき＆実在を検査。空は許可）。
- `Modify: tests/test_hwlint.py` — ①derived-from 接頭辞なし→error ②存在しない派生元→error ③正しい派生元→pass ④空→pass。

## 要判断（決めてから実装）

### P4: sources/ の編集権ポリシーを一本化する

**食い違い**:
- `CLAUDE.md` 3層表: sources は「人間が置く。AIは**読むだけ**。編集・削除禁止」。
- `.claude/skills/ingest/SKILL.md` 手順1: 「まだ置かれていなければ AI が sources/ に保存する」。
- `tools/hooks/guard_sources.py`: 「新規ファイルの Write は許可・既存の編集は禁止」。

実装（ingest＋guard）が事実上の仕様になっている。今回の WebSearch 言説も AI が sources に新規保存した。

**選択肢**:
- **(A) 新規追加を正式に認める（推奨）**: CLAUDE.md 3層表を「AIは読むだけ（既存は不変）。ただしインタビュー録等の**新規生データの追加は可**」に精緻化。ingest/guard は現状維持。→ 実務（ingest が録を保存）と一致。
- **(B) AI の sources 書き込みを全面禁止**: ingest 手順1を「生データは人間が置く」に変更、guard_sources を新規 Write もブロックに。→ CLAUDE.md 現行文言に忠実だが、ingest の実運用が窮屈に。

**推奨**: (A)。決定後 `CLAUDE.md`（3層表・不変ルール3）を1文修正するだけ（コード変更なし。guard は現状で(A)準拠）。

### P2: ingest の確信度反映に非対話モードを設ける

**問題**: ingest は「確信度・ステータスは必ずユーザー承認を得てから反映」。自律スモーク・将来のバッチ/CI 運用では各更新で止まると回らない（今回は提案明示のうえ自己承認で進めた）。

**選択肢**:
- **(A) 現状維持（常に承認）**: 規律最優先。自律運用は非対応のまま。
- **(B) 承認要否を段階化（推奨）**: 事前確定した成功基準に対し判定が機械的に〈支持/反証〉に定まる場合は自動反映可、〈判断保留〉や成功基準の解釈を要する場合のみ承認を求める。ingest/SKILL.md 手順4-5に「非対話モードの既定挙動」を追記。
- **(C) 明示フラグ運用**: 「一括反映してよい」と人が明示したセッションでのみ自動反映。

**推奨**: (B)。コードなし（SKILL.md の文言）。ただし「証拠の階梯を超えて上げない」既存規律と両立させる。

### P3: Stop フックの中間状態ブロックを緩めるか

**再評価**: `stop_lint.py` は `stop_hook_active` ガードにより、error があっても**そのターン終了を1回ブロックして修正を促し、次の停止では素通し**する（無限ループはしない）。よって「詰まる」より「1回ナッジする」挙動。当初想定より深刻度は低い。

**残る懸念**: 編集途中（frontmatter を変えたが履歴未追記など）の一時的 error でも1回ナッジが入る。

**選択肢**:
- **(A) 現状維持（1回ブロック）**: 完了漏れを確実に捕捉。中間状態では1回だけ余分なナッジ。
- **(B) 助言化（systemMessage・exit 0）**: error を表示するだけでブロックしない。強制はコミット時(pre-commit)に一本化。
- **(C) 未コミット変更がある間は猶予**: 「作業中」を検出して見送り、クリーンな状態でのみブロック。

**推奨**: (A) 現状維持で十分（再評価で深刻度が下がったため）。頻繁に邪魔になるなら (B) に切り替え。この項目は「決めて据え置き」で可。
