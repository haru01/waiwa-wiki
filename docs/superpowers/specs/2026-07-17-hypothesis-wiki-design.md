## ユーザーと合意済みの設計判断（変更しないこと）

1. **用途**: 再利用可能なキット＋実例の両方。ドッグフーディング題材は**このツール自体**（誰のどんな課題を解決するかをこのWikiで検証する）
2. **確信度**: **10段階**（1〜10）。アンカー: 検証済みで確信度が高い=8、事実=10。別軸で**ステータス**（未検証/検証中/検証済み/反証）を管理
3. **記録は3層＋テストカード構造**: log.md（追記専用1行タイムライン）／活動レコード（Strategyzerのテストカード＆学習カード構造：検証前に成功基準を固定）／意思決定レコード（ADR風、確信度スナップショットと巻き戻しポイント付き）
4. **キャンバスは保存せずビューで**: 仮説レコードがSSoT。**バリュープロポジションビュー**と**一覧ビュー**の2つを生成（ジャベリンボード風ビューはオプション）
5. **重要度**: 現在ステージから自動決定（ステージ→重点仮説タイプのマッピングをスキーマに定義）＋仮説ごとの手動上書き可
6. **確信度の更新は必ず /ingest か /decide 経由**（活動か意思決定に紐づく）。AIが根拠なく確信度を変えない
7. 記述言語は日本語

## スキーマの要点（CLAUDE.md に書く内容）

- **仮説レコード** frontmatter: `id: H-NNN` / `title` / `type: 状況・行動仮説|課題仮説|ソリューション仮説|買ってもらえる仮説` / `status: 未検証|検証中|検証済み|反証` / `confidence: 1-10` / `stage: CPF|FPF|PSF|SPF|PMF`（検証対象ステージ）/ `importance: auto|1-10`（省略時はステージから自動）/ `derived-from: H-NNN`（派生元。ピボット・分割・巻き戻し後の再出発の系譜）。本文: 仮説文（反証可能な形式）、前提、系譜（`[[H-NNN]]`リンク）、確信度履歴テーブル（日付・確信度・ステータス・根拠・活動リンク）
- **活動レコード** frontmatter: `id: ACT-NNN` / `type: interview|demo|survey|mvp-test|desk-research` / `date` / `stage` / `hypotheses: [H-NNN…]`。本文: 【テストカード】目的・方法・指標・成功基準（検証前に記入）→【学習カード】事実（observed）／解釈（inference）／驚き・想定外／確信度の更新テーブル／次のアクション
- **意思決定レコード** frontmatter: `id: DEC-NNN` / `date` / `type: stage-transition|pivot|persevere|rollback|kill` / `based-on: [ACT-NNN…]`。本文: 確信度スナップショット、選択肢と判断理由、巻き戻しポイント
- **確信度10段階の目安表**（1-2:勘 / 3-4:二次情報 / 5-6:検証中で手応え / 7-8:検証済み高確信 / 9:反証に耐えた / 10:事実）
- **ステージ→重点仮説タイプ**: CPF→状況・行動＋課題、FPF→課題×自分たちの取り組む理由、PSF→ソリューション、SPF→ソリューション＋買ってもらえる、PMF→買ってもらえる＋市場全体
- **不変ルール**: sources/は編集不可。log.mdは追記のみ。views/は再生成可能な生成物。確信度変更はACT/DECに必ず紐づける。ID採番は既存最大値+1

## Obsidian連携（探索ネットワークの可視化）

探索の履歴は「仮説の系譜（派生・ピボット・巻き戻し）」と「仮説↔活動↔意思決定の参照」でツリー／ネットワーク状になる。これをObsidianのグラフビューでそのまま見られるようにする：

- **ファイル名はIDそのもの**（`H-021.md`、`ACT-012.md`、`DEC-005.md`）。タイトルはfrontmatter `title`と本文H1に持つ。これで `[[H-021]]` 形式のwikilinkが常に解決され、リネームによるリンク切れが起きない
- **相互参照は必ず本文にwikilinkで書く**規約（frontmatterの配列だけではObsidianのグラフに辺として現れないため）。ACTは対象仮説へ、DECは根拠ACTと影響仮説へ、仮説は派生元 `[[H-NNN]]` へリンクする
- リポジトリルートをObsidian vaultとして開く。フォルダ（hypotheses/activities/decisions/sources）ごとにグラフの色分けができ、仮説ネットワーク・検証の足跡・意思決定の分岐が一望できる
- `.obsidian/` は.gitignoreに入れる。README にvaultとして開く手順とグラフビュー活用（views/をフィルタで除外する等）を記載
- frontmatterはDataviewでの動的テーブル（確信度一覧など）にもそのまま使える形にする

## ステージプレイブック（playbooks/ 各ファイルの内容）

各ステージにつき「中心の問い（問いかけバンク）／形成すべき仮説タイプ／適した検証手法／次ステージへの移行基準」の4点セットを定義する。`/grill`（問いかけ）・`/plan`（検証手法の提案）・`/decide`（移行基準の判定）がこれを参照する。

| ステージ | 中心の問い（例） | 重点仮説タイプ | 適した検証手法 | 移行基準（例） |
|---|---|---|---|---|
| **CPF** | 想定顧客は誰か？どんな状況でどんな行動をとるか？その課題は実在し、顧客は自覚しているか？ | 状況・行動仮説、課題仮説 | 問題インタビュー、行動観察、デスクリサーチ | 複数顧客が同一課題を自認し課題仮説が確信度7以上 |
| **FPF** | なぜ自分たちがこの課題に取り組むのか？原体験・強み・持続する情熱はあるか？ | 課題仮説×自分たち（創業者適合）仮説 | 原体験の棚卸し、チーム対話、小さくコミットして継続するか観察 | 取り組む理由が言語化されチームが納得している |
| **PSF** | この課題はこのソリューションで解決するか？顧客はそれを欲しがるか？ | ソリューション仮説 | ソリューションインタビュー、ペーパープロトタイプ、モックデモ | プロトタイプに対して顧客が価値を認め対価に言及 |
| **SPF** | ソリューションをプロダクトとして提供できるか？使い続けてもらえるか？ | ソリューション仮説＋買ってもらえる仮説（前半） | MVP、クローズドβ、利用データ計測 | 継続利用が観測され支払い意向が確認できる |
| **PMF** | 市場はこれを求めているか？スケールするか？ | 買ってもらえる仮説、市場仮説 | 実販売、リテンション計測、Sean Ellisテスト | 定量指標（継続率・「なくなったら困る」40%等） |

問いかけバンクは各playbookに10問程度ずつ用意し、`/grill`はここから未回答・未仮説化の問いを選んで1問ずつ聞く。移行基準はテンプレートであり、プロジェクトごとに`wiki/stage.md`で上書き調整できる。

## スキル仕様の要点（各SKILL.md）

| スキル | 動作 |
|---|---|
| `/grill` | 現在ステージのプレイブックの問いかけバンクから、未回答・未仮説化の問いを選んで1問ずつ質問（AskUserQuestion活用）→ 曖昧なアイデアを反証可能な仮説文に精錬 → タイプ判定・初期確信度/ステータス設定 → H-NNN発番、hypotheses/作成、index.md・log.md更新 |
| `/plan` | stage.md＋index.mdを読み、重点タイプ×低確信度×未検証/検証中の仮説を抽出 → 重要度×証拠の2軸マップ（アサンプションマッピング）を提示 → プレイブックの「適した検証手法」から方法を提案し、選ばれた仮説のテストカード（ACT草稿、検証前セクションのみ）を作成 |
| `/ingest` | 生データをsources/へ → 対応ACTの学習カード記入（事前ACTがなければ新規作成） → 事実/解釈/驚きを分離して記録 → 確信度・ステータス更新を**提案しユーザー承認後に**仮説レコードと履歴テーブルへ反映 → log.md追記 |
| `/view` | サブコマンド `vp`（顧客プロファイル×価値マップの対応表示）/ `list`（全仮説テーブル）/ `board`（ジャベリンボード風・実験の変遷）を wiki/views/ に生成 |
| `/decide` | プレイブックの移行基準（stage.mdの上書きがあればそちら）に照らして現ステージの達成度をチェック、確信度スナップショット生成、選択肢比較 → DEC作成、stage.md更新、log.md追記。`rollback`時は過去DECの巻き戻しポイントから復元手順を提示 |
| `/lint` | 矛盾する仮説、孤立レコード、証拠リンクのない「検証済み」、長期放置の「検証中」、index/logの不整合を検出してレポート |

## 実装ステップ

0. `git init` → 本計画書を `docs/superpowers/specs/2026-07-17-hypothesis-wiki-design.md` としてリポジトリに保存し最初のコミット → `gh repo create hypothesis-wiki --private --source . --push` で**privateリポジトリ**としてGitHubへpush（gh認証状態を確認し、必要ならユーザーに `! gh auth login` を依頼）
1. 以降、ステップごとにコミットしてpushする
2. CLAUDE.md（スキーマ層）を書く — 上記「スキーマの要点」を全て含む
3. playbooks/ 5ファイルを書く（各ステージ：問いかけバンク10問程度＋重点仮説タイプ＋検証手法＋移行基準）
4. templates/ 3ファイルを書く（スキーマと完全整合させる）
5. .claude/skills/ 6スキルを書く（frontmatter: name/description、descriptionは発動条件を具体的に）
6. wiki/ 初期ファイル（index.md・log.md・stage.md=CPF、空ディレクトリ＋.gitkeep）と sources/README.md、.gitignore（.obsidian/）
7. README.md（キットとしての使い方：別案件への複製手順＝wiki/とsources/を空にしてコピー、Obsidian vaultとして開く手順）
8. コミット・push後、**ドッグフーディング開始**: `/grill` をこのツール自体に適用し、最初の仮説群（例：「開発チームは仮説検証の記録を残せていない」等の課題仮説）を数件記録 → `/view list` で一覧生成 → 実例としてコミット・push

## 検証方法（end-to-end）

1. `/grill` を実行し、対話から H-001〜 が正しいfrontmatterで生成され、index.md・log.mdに反映されること
2. `/plan` が「CPFステージ×課題仮説×低確信度」を優先抽出し、テストカード付きACT草稿を作ること
3. 架空のインタビューメモをsources/に置いて `/ingest` → 学習カード・確信度更新（承認フロー付き）・log追記が起きること
4. `/view vp` `/view list` が記録と矛盾しないビューを生成すること
5. `/lint` が意図的に仕込んだ不整合（証拠リンクなしの検証済み仮説）を検出すること
6. `grep "decision" wiki/log.md` 等でタイムラインが機械抽出できること
7. リポジトリをObsidian vaultとして開き、グラフビューで H↔ACT↔DEC のネットワークと仮説の系譜が辺として表示されること（本文wikilink規約の確認）
