# 設計: `/prototype` スキル — 検証用HTMLプロトタイプ生成

- 日付: 2026-07-18
- ステータス: 承認済み（実装計画へ）
- 対象: `.claude/skills/prototype/SKILL.md` の新規追加

## 目的

検証対象の仮説（H）から、インタビュー／デモで見せるクリッカブルな **HTML＋JavaScript 自己完結プロトタイプ**
（ランディングページ、または 2〜3画面モックアップ）を「サクッと」生成し、`demo`/`interview` の
活動レコード（ACT）に紐づける。生成物は検証活動の小道具であり、仮説→ACT→プロトタイプの
トレーサビリティを Wiki 上に残す。

## 決定事項（ブレインストーミングで確定）

| 論点 | 決定 |
|---|---|
| Wikiとの結合度 | **密結合**。検証対象の仮説を読み、価値提案・課題・CTAを反映。ACT（demo/interview）に紐づける |
| 主な検証方法 | **デモ・インタビューで見せる**（定性フィードバック中心）。実データ計測コードは不要 |
| 置き場所 | `projects/<slug>/wiki/prototypes/<PREFIX>-ACT-NNN/index.html`。ACT にフォルダ紐づけ |
| ファイル構成 | LPは単一ページ・**世間標準のSaaS系構成**（ナビ→ヒーロー＋SVGイラスト→トラスト→課題→特徴→使い方→声→料金→最終CTA→フッター）。イメージは**インラインSVG**。声・料金は「デモ用ダミー／仮の表示」と明示。モックアップは**Webアプリ**を想定（上部バー＋左サイドバーナビ＋メインのシェル。スマホアプリ枠にしない）。**単一HTML内 `<section>`＋JS画面切替を基本**、必要時のみ分割（スキル判断） |
| ACTの責務 | **両対応**。計画済みの demo/interview ACT があれば紐づけ、無ければ最小テストカードをその場で作成 |
| インテイクの重さ | **リーン型**。仮説から自動ドラフト → 最小確認 → 一発生成 → 対話で微修正。定番レイアウト骨格を内包し品質担保 |

## スキル仕様

### メタ

- **名前**: `/prototype`
- **description トリガー語**: 「モックアップ」「プロトタイプ」「LP」「ランディングページ」「画面を作って」
  「prototype」「mockup」「mock」「lp」

### ワークフロー

1. **現在プロジェクト解決** — `projects/current.md` の `current-project: <slug>` を読み、以降 `wiki/` は
   `projects/<slug>/wiki/` を指す。接頭辞（例 `SELF`）もここから決まる。
2. **検証対象の仮説（H）の選択** — 引数で ID 指定があればそれ。無指定なら「次に検証すべき仮説」
   （重要度高 × 確信度低 × ステータス未検証/検証中）を数件提示して選ばせる。
3. **種別選択** — LP か モックアップ か（引数 or 対話）。
4. **最小インテイク（2〜3点）** — 対象仮説と関連レコード（課題仮説・ソリューション仮説・買ってもらえる仮説など）
   から内容を**自動ドラフト**し、ユーザーは確認・上書きするだけ。
   - LP: ヒーロー見出し（価値提案）／対象顧客／主要ペイン／CTA文言
   - モックアップ: 見せたい画面（2〜3）とその主目的
5. **ACT解決（両対応）**
   - 対象仮説に紐づく **計画済みの demo/interview ACT**（テストカードのみで学習カード未記入のもの）が
     あれば、それに紐づける。
   - 無ければ **最小テストカード**（目的／方法／指標／成功基準）を新規作成。ID採番は
     種別×プロジェクトで既存最大+1・接頭辞つき（例 `SELF-ACT-005`）。`templates/activity.md` に従う。
     type は `demo` または `interview`。
6. **生成** — `projects/<slug>/wiki/prototypes/<PREFIX>-ACT-NNN/index.html`
   - 自己完結: インラインCSS/JS、外部依存ゼロ（CDN・外部フォント・fetch・画像URL無し）。
     アイコンは絵文字/インラインSVG、写真はプレースホルダ。`file://` でダブルクリック起動できる。
   - レスポンシブ、日本語UI。
   - モックアップは単一HTML内 `<section>`＋JSで show/hide 画面切替を基本。画面数・複雑さ次第で分割可（スキル判断）。
   - CTAボタンやフォームは、クリックするとクライアント側で確認状態（サンクス表示など）を出すだけで、
     送信先（バックエンド・外部フォーム）は持たない。デモ・インタビューで反応を見るための小道具として振る舞う。
   - HTML先頭に生成物メタコメント: 生成物である旨・紐づく `ACT`/`H` のID・生成日。
7. **記録**
   - ACTレコードのテストカードにプロトタイプへの相対リンク（`../prototypes/<PREFIX>-ACT-NNN/index.html`）を追記。
   - 仮説レコード本文に `[[<PREFIX>-ACT-NNN]]` を追記（frontmatter配列だけでなく本文wikilink）。
   - `wiki/log.md` に1行追記（type: `demo` または `interview`）。追記のみ・過去行は編集しない。
8. **反復** — 生成後、見た目・文言の微修正を対話で受け、同じ `index.html` を上書き再生成する。

### 守ること（規約整合）

- **確信度・ステータスはこのスキルでは変更しない**。プロトタイプ作成は検証*前*の準備活動である。
  実際の検証（見せて反応を得る）後の学習カード記入・確信度更新は **`/ingest` に委ねる**（責務分離）。
  → ACT はテストカード（前半）だけの状態で作られ、これは「テストカードは検証前に記入」規約と一致する。
- テストカードの成功基準は検証開始後に書き換えない（後知恵バイアス防止）。
- `sources/` は読むだけ。`wiki/prototypes/` は生成物。`log.md` は追記のみ。
- ID採番・プロジェクト接頭辞・wikilink規約（本文に `[[...]]`、schema層は相対mdリンク）に従う。
- frontmatter `id` はファイル名と完全一致（接頭辞つき）。

### 生成HTMLの品質方針

- 外部依存ゼロで `file://` から開ける。CSP的にも自己完結。イメージ図はインラインSVG。
- インタビューで「本物っぽく」見える最小限の作り込み。過剰装飾しない。
- レイアウト骨格は schema層の雛形として `templates/prototype-lp.html`・`templates/prototype-mockup.html` に置き、生成時に読み込んで `{{...}}` を差し替える（`/plan` が `templates/activity.md` 等を参照するのと同じ方式）。構成は上の決定事項表のとおり（LP＝SaaS系フル構成＋SVG／モックアップ＝Webアプリシェル）。骨格は品質の下限を担保するもので、仮説内容に応じて中身は差し替える。

## `/plan`・`/ingest` との責務分担

| スキル | 役割 |
|---|---|
| `/plan` | 次に検証すべき仮説の抽出とテストカード立案（計画）。ACTのテストカードを起こせる |
| `/prototype` | 検証の**小道具（HTMLプロトタイプ）を作る**。ACTが無ければ最小テストカードも起こす（両対応） |
| `/ingest` | 検証**後**の取り込み。学習カード記入・確信度/ステータス更新はここが担う |

`/prototype` は `/plan` と `/ingest` の中間に位置する。テストカード立案（前半）までは `/plan` と重なりうるが、
`/prototype` は「見せる成果物の生成」が主目的で、ACT作成は不足時の補完として行う。

## スコープ外（YAGNI）

- 実データ計測・分析コード（バックエンド無し前提。デモ・インタビュー用途では不要）。
- 外部フォーム連携（Google フォーム/Tally/mailto）。将来スモークテスト用途を足すときの拡張余地として残す。
- 複数プロジェクト横断のプロトタイプ管理。現在プロジェクトのみ対象。
- ビュー（`/view`）による自動集計対象化。プロトタイプHTMLはレコードから乖離しうる生成物として扱い、
  `/view` の対象にはしない（既存の `index.html` 扱いと同じ方針）。

## 成功基準

- `/prototype` 実行で、対象仮説を選び、LP/モックアップを1つ生成できる。
- 生成HTMLが `file://` でダブルクリック起動し、モックアップは画面切替が動く。
- ACTレコード（テストカード）・仮説本文wikilink・log.md が規約どおり更新される。
- 確信度・ステータスは変更されない（`/ingest` の担当領域を侵さない）。
