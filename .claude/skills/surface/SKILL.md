# /surface — 目的仮説の種を引き出す（探索）

漠然としたキャリア観から、目的仮説（P）の種を対話で引き出し、**低い確信度（1-2）で起票する**。目的は事前に見つけるものでなく試行錯誤で立ち上がるので、ここでは確信を高めようとしない。広げるフェーズ。

> **共通規約**（案件解決・ID/接頭辞・リンク記法と `../` 深さ・.gitkeep・承認規律）は [CLAUDE.md「スキル共通規約」](../../../CLAUDE.md) が正典。以下は本スキル固有の手順。

## 手順

1. **現在案件を読む**（→ 共通規約1）。[playbooks/formation.md](../../../playbooks/formation.md) の「問いかけバンク A（種を引き出す）」を読む。`wiki/index.md` で既存の目的仮説を把握し、まだ言語化されていない領域を探す。

2. **1問ずつ引き出す** — `AskUserQuestion` で問いかけバンク A を**1問ずつ**。過去の事実（没頭したこと・放っておけないこと・やってよかった場面）を具体で聞く。抽象的な「なりたい像」に流れたら「直近の具体例は？誰がどう反応した？」で事実に降ろす。**このセッションのログは壁打ち（dialogue）** なので、それ自体では確信度を上げない（外界に触れていない）。

3. **種を目的仮説 P として起票する** — 反証可能な一文に近づいたら `wiki/purposes/` の既存最大+1で `P-NNN` を採番し `templates/purpose.md` で作成する。
   - `confidence: 1-2`（勘・思いつき）、`status: 未検証`。
   - `id` はファイル名と同じ接頭辞つき（例 `SELF-P-001`）。`short-title` に短ラベル。
   - まだ一文に整わなければ、無理に反証可能化せず「種」として本文にメモし、次に `/formulate-purpose` で精錬する旨を残す（精錬はあちらの役割）。
   - 確信度履歴テーブルに初期行（`初期作成（勘・思いつき）`・活動列 `—`）を入れる。`.gitkeep` が残っていれば削除してよい。

4. **停滞時の幅方向探索（再起動）** — `/lemonade`（④レモネード）・`/pilot`（⑤）や `/stall-check`（フェーズ2）から「探索域を開き直す」ために呼ばれたときは、**いま確信が一本化しつつある P から意図的に離れた領域**で問いを立てる（「その目的“ではない”としたら、手の内にある別の出発点は？」）。生まれた種は既存 P と別系統として起票し、必要なら `derived-from` で分岐元を示す。`wiki/explore-log.md` に `探索再起動` を1行追記する。

5. **index と log を更新する** — `wiki/index.md` の目的仮説表に1行追加。`wiki/log.md` に追記:
   `## [YYYY-MM-DD] purpose | P-NNN <要約> → 新規作成 確信度N/未検証`

## 守ること

- 一度に大量生産しない。1本ずつ。低い確信度を恐れず正直に付ける（種は種でよい）。
- **壁打ちで確信度を上げない**（証拠の非対称性）。ここは広げる場。確信を育てるのは `/reflect`（外界の証拠）。
- 「探索の永久機関化」を避けるため、種を増やしっぱなしにせず、有望なものは `/formulate-purpose`→試行 へ送る。
