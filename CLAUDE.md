# キャリア目的形成Wiki — スキーマ

このリポジトリは、試行錯誤の ReAct ループを通じて**キャリアの目的・制約・使命が本人の中に事後的に立ち上がること**を支援する個人向け LLM-wiki である。日々の壁打ち・試行を `sources/` に沈殿させて**確信を育て**、停滞したら**意図的に確信を揺らして探索域を開き直す**。AIはこのファイルの規約に従って「規律あるWikiの保守者」として振る舞う（触媒・鏡・問い直す役〔`/pilot`〕であって、目的を代わりに決める者ではない）。

## 案件（人単位）

目的形成は**案件（人）単位**で分ける。各案件は `projects/<slug>/` 配下に自分の `sources/`（生データ）と `wiki/`（生成・保守層）を持つ。スキーマ層はリポジトリ全体で共有する。現在アクティブな案件は `projects/current.md` の `current-project: <slug>` が持つ。スキルはまず `projects/current.md` を読み、`projects/<slug>/` 配下を対象に動く。

以下このスキーマで `sources/` `wiki/` と書くときは、断りがなければ**現在の案件の `projects/<slug>/sources/`・`projects/<slug>/wiki/`** を指す。

## 3層アーキテクチャ

| 層 | 場所 | 編集権 |
|---|---|---|
| Raw Sources（不変層） | `projects/<slug>/sources/` | 人間または `/reflect` が生データを置く。AIは**既存ファイルを改変しない**（新規追加は可・既存の編集禁止）。冒頭に**種別タグ**（trial/dialogue/third-party）を必ず書く |
| The Wiki（生成・保守層） | `projects/<slug>/wiki/` | AIが規約に従って作成・更新する |
| The Schema（設定層） | `ontology.yaml`（型・関係の正本）・`CLAUDE.md`・`AGENTS.md`・`playbooks/`・`templates/`・`.claude/skills/` | 人間が合意の上で変更する（全案件共有） |

## オントロジー（型・関係の正本）

レコードの**型**（エンティティ P/C/ACT/REF/DEC とサブタイプ）、レコード間の**型付きリンク**（関係）、形成の**状態機械**（ステータス・確信度・証拠の階梯）は、[ontology.yaml](ontology.yaml) が唯一の正本（SSoT）である。人間可読な要約は [ontology.md](ontology.md)（`python3 tools/gen_ontology_doc.py` で生成・手編集禁止）。ツール（`tools/hwlint.py`・`tools/gen_views.py`）は `tools/ontology.py` 経由でここを読むため、**語彙(enum)・関係・状態機械をコードや本CLAUDE.mdに再定義しない**（二重管理・ドリフト防止）。

**レコード種別**は5つ:
- **P（目的仮説）** — 立ち上げつつある目的。確信度（1軸目）を持つ。
- **C（制約・手中の鳥）** — Effectuation の所与の資源（自分は誰か／何を知っているか／誰を知っているか）。
- **ACT（試行）** — 外界への働きかけ（ReAct の Act。打診／やってみる／面談／観察／self-reflection）。
- **REF（内省）** — ReAct の Reason/Observe（事実／解釈／驚き）。確信度は持たない。
- **DEC（意思決定）** — ピボット・巻き戻し・棚上げ・意図的引き下げ・モード切替。

**関係（型付きリンク）** は7種。各々 domain→range・cardinality・inverse を `ontology.yaml` の `relations` で宣言する。`derived-from`（P→P・派生元）／`leads-to`（P→P・因果先）／`grounded-in`（P→C・接地する制約）／`revises`（P→P・書き換え元）／`purposes`（ACT→P・検証対象）／`reflects-on`（REF→P・内省対象）／`based-on`（DEC→ACT・根拠活動）。関係は frontmatter 配列と本文 wikilink の**二重表現**を持つ。

## スキル共通規約（全スキルが従う入口）

`.claude/skills/` の各スキルは、冒頭でこの節を参照し**そのスキル固有の手順だけ**を書く（下記の規約を各スキルにコピーしない）。

1. **案件解決** — まず `projects/current.md` の `current-project: <slug>` を読み、「プロジェクト一覧」表で接頭辞（PREFIX）を確定する。以降 `sources/` `wiki/` は `projects/<slug>/` 配下を指す。`/lint` `/view` は現在案件のみを対象にする。
2. **ID・接頭辞** — ID＝ファイル名＝frontmatter `id` を三者一致させ、すべて案件接頭辞つき（例 `SELF-P-001`）。採番は種別×案件ごとの既存最大+1。再利用禁止（取り下げた番号は欠番として残す）。ID の infix は P/C/ACT/REF/DEC。
3. **リンク記法** — 接頭辞つきノート間の相互参照は**必ず本文に wikilink**（`[[SELF-P-001]]`）。schema層（`playbooks/`・`CLAUDE.md` 等）は**相対mdリンク**で書く。`../` の深さは参照元ファイルの位置で変わる:

   | 参照元の位置 | 深さ | 例 |
   |---|---|---|
   | `wiki/` 直下（`stage.md`・`index.md`・`explore-log.md`） | `../../../` | `[playbooks/formation.md](../../../playbooks/formation.md)` |
   | `wiki/<種別>/` 配下（P・C・ACT・REF・DEC） | `../../../../` | `[playbooks/formation.md](../../../../playbooks/formation.md)` |
4. **.gitkeep** — 空ディレクトリ雛形の `.gitkeep` は、そのディレクトリに最初のレコードを作成したら削除してよい（任意）。
5. **承認規律** — 確信度・ステータスの変更は必ず ACT/DEC に紐づけ、**提案 → ユーザー承認 → 反映**する（不変ルール参照）。非対話/バッチ実行では、①判定が機械的に定まり、②提案する確信度が証拠の階梯（下記）の範囲に収まる場合に限り、提案内容を明示のうえ自動反映してよい。解釈を要する／証拠の階梯を超える引き上げは、必ず対話で承認を得る。

## レコード種別とスキーマ

すべてのレコードは `templates/` の雛形に従う。ファイル名は**IDそのもの**で、**案件接頭辞つき**（例 `SELF-P-001.md`）。frontmatter の **`id` はファイル名と完全に一致させる**。相互参照は**必ず本文に wikilink**で書く。

### 目的仮説 `wiki/purposes/<PREFIX>-P-NNN.md`

```yaml
id: <PREFIX>-P-001
title: 短いタイトル
short-title: 短ラベル                 # 省略可
status: 未検証 | 探索中 | 立ち上がりつつある | 立ち上がった | 棚上げ
confidence: 1-10                     # 目的の硬さ
# engagement-temp: 1-10              # 当事者性温度。フェーズ1では予約・未使用（記入しない）
derived-from: <PREFIX>-P-NNN         # 省略可・単一。派生・ピボット・巻き戻しの系譜
revises: <PREFIX>-P-NNN              # 省略可・単一。前言撤回（旧目的の書き換え）
leads-to: [<PREFIX>-P-NNN, ...]      # 省略可。因果的に導く先の目的
grounded-in: [<PREFIX>-C-NNN, ...]   # 省略可。根ざす制約（手中の鳥）
core: true                           # 省略可。核心目的なら true
```
本文: 反証可能な目的仮説文／前提／系譜・接地リンク／確信度履歴テーブル（日付・確信度・ステータス・根拠・`[[ACT/DEC-NNN]]`）。

### 制約・手中の鳥 `wiki/constraints/<PREFIX>-C-NNN.md`

```yaml
id: <PREFIX>-C-001
title: 短いタイトル
type: 自分は誰か | 何を知っているか | 誰を知っているか
```
本文: 制約の記述／これがどう目的の出発点になるか。

### 試行 `wiki/activities/<PREFIX>-ACT-NNN.md`

```yaml
id: <PREFIX>-ACT-001
title: 短いタイトル
type: 打診 | やってみる | 面談 | 観察 | self-reflection
date: YYYY-MM-DD
purposes: [<PREFIX>-P-NNN]
riskiest-assumption: 一文               # 最もリスクの高い前提（この試行で崩れたら目的が崩れる一点）。検証前に記入
outcome: 起票|支持|反証|判断保留|是正      # 検証後に /reflect が記入（検証前は空＝未実施）
```
本文は2部構成（Strategyzer流）: **テストカード**（検証前・後から書き換えない）: 目的／方法／指標／成功基準。**学習カード**（検証後）: 学びの要点／事実（observed）／解釈（inference）／驚き・想定外／確信度の更新テーブル／次のアクション。

### 内省 `wiki/reflections/<PREFIX>-REF-NNN.md`

```yaml
id: <PREFIX>-REF-001
title: 短いタイトル
date: YYYY-MM-DD
reflects-on: [<PREFIX>-P-NNN]
source: sources/YYYY-MM-DD-....md      # 省略可
```
本文: 事実／解釈／**驚き**。内省は確信度を持たない（確信度を動かすのは ACT/DEC）。「驚き」が枯れているのは停滞のシグナル（フェーズ2の停滞検知の入力）。

### 意思決定 `wiki/decisions/<PREFIX>-DEC-NNN.md`

```yaml
id: <PREFIX>-DEC-001
title: 短いタイトル
date: YYYY-MM-DD
type: モード切替 | ピボット | 継続 | 巻き戻し | 棚上げ | 意図的引き下げ
based-on: [<PREFIX>-ACT-NNN]
```
本文: 確信度スナップショット／選択肢と判断理由／巻き戻しポイント／次の一手。

## 二つの温度（本ツールの中核概念）

本ツールは**二つの独立した量**を扱う。混同すると設計が崩れる。

1. **確信度（1–10）＝目的の硬さ** — その目的仮説がどれだけ「これだ」と感じられているか。**証拠に紐づけてのみ動く。**
2. **当事者性温度＝関わりの深さ** — その目的にどれだけ腰を据えて関わっているか。**フェーズ1では実装しない**（`engagement-temp` を予約するのみ・状態機械で検証しない・ビューに出さない・本人にも見せない＝ゲーム化防止）。停滞検知・ハンドオフを作るフェーズ2以降で本実装する。

確信度が高くても当事者性が低い（頭では固まったが自分ごとになっていない）状態がありうる——これが本ツールが最も警戒する**偽の収束**。フェーズ1では確信度＋証拠の非対称性で運用し、当事者性の観測手がかりは実運用ログから集める。

## 確信度と証拠の非対称性

**確信度（1〜10）** — 外界に触れた証拠の強さの目安:

| 確信度 | 目安 |
|---|---|
| 1-2 | 勘・思いつき（壁打ちで言語化した段階） |
| 3-4 | 探索中。手応えの兆しはあるが外界の証拠は薄い |
| 5-6 | 試したら反応があった（〈試行〉以上） |
| 7-8 | 他者が実際に動いた／繰り返し起きている（〈他者反応〉〈継続〉） |
| 9-10 | 揺さぶっても崩れなかった／観測された確定事項 |

**ステータス**: `未検証` → `探索中` → `立ち上がりつつある` → `立ち上がった` ／ `棚上げ`

**証拠の階梯**（外界にどれだけ触れたか。弱→強。正本は [ontology.md](ontology.md) の `evidence-ladder`）:

〈壁打ち〉エージェントとの対話のみ ＜ 〈試行〉自分で小さくやってみた ＜ 〈他者反応〉第三者が実際に動いた ＜ 〈継続〉繰り返し起きている

- **証拠の非対称性（本ツールの要）**: 確信度を**上げてよいのは〈試行〉以上（＝外界に触れた証拠）に紐づくときだけ**。〈壁打ち〉（対話のみ）単体では上げない。壁打ちログは目的文を**磨く**（`revises`）・確信度を**下げる**のには自由に使う。この非対称は `hwlint.py` の `check_evidence_asymmetry` が機械強制する。
- 確信度 5-6 には〈試行〉以上、7-8 には〈他者反応〉以上を要する。
- **架空/脳内シミュレーション由来の確信度は上限8**。9-10 は実観測に限る。
- 確信度履歴の「根拠」列は、先頭に証拠種別タグを付ける（例 `〈他者反応〉3人中2人が次の約束をした`）。
- source には**種別タグ**（trial/dialogue/third-party）を冒頭に必ず書く。dialogue 由来だけでは確信度を上げられない。

### 不変ルール（AIが必ず守ること）

1. **確信度・ステータスの変更は必ず活動（ACT）か意思決定（DEC）に紐づける**。根拠レコードなしに書き換えない。
2. **確信度を上げるのは外界に触れた証拠（〈試行〉以上）があるときだけ**。壁打ち（dialogue）単体では上げない（証拠の非対称性）。
3. 変更時は目的レコードの確信度履歴テーブルに1行追記し、`wiki/log.md` にも追記する。
4. `sources/` の既存ファイルは改変・削除しない（新規追加は可）。`wiki/log.md`・`wiki/explore-log.md` は追記のみ。
5. `wiki/views/` は生成物。記録の修正はレコード側で行い、ビューは再生成する。
6. ID採番は種別×案件ごとに既存最大+1で接頭辞つき。再利用禁止（取り下げた番号は欠番）。
7. テストカードの成功基準は検証開始後に書き換えない（後知恵バイアス防止）。

## 探索モードと停滞検知

目的形成は「**深掘り（exploit・確信を高める）**」と「**探索（explore・探索域を開き直す）**」の二相を往復する。現在のモードは `wiki/stage.md` の `current-mode`（探索/深掘り）が持つ。往復のトリガーは**停滞検知**。

- **フェーズ1（いま）**: 停滞検知は**手動**。週次で `wiki/explore-log.md` と `/view` を見て、深掘りに偏って停滞していないか本人が判断する。停滞を感じたら `/pilot`（⑤・深さ方向＝既存Pの確信度を疑い引き下げ）か `/lemonade`（④・幅方向＝驚き/偶然を逆手に新しい種）／`/surface` 再起動で探索域を開き直し、`explore-log.md` に記録する。
- **フェーズ2以降**: `/stall-check` が停滞シグナル（新規試行の枯渇／確信度の高原／驚きの枯れ／系譜の停止）を自動評価する。ただし**確信度を自動では下げない・必ず本人承認を挟む**。当事者性温度をここで本実装し、偽の収束（確信高×当事者性低）と真の立ち上がり（両方高）を判別する。

## log.md の形式（追記専用・grep可能）

```
## [YYYY-MM-DD] <type> | <ID> <要約> → <影響目的と確信度変化>
```
type は `purpose` `constraint` `打診` `やってみる` `面談` `観察` `self-reflection` `reflection` `decision` `lemonade` `pilot` `lint` のいずれか。確信度変化は `P-001 1→5` のように書く。

## ワークフロー（スキルとの対応）

キャリア目的形成のループは**エフェクチュエーション（Effectuation）の5原則**を骨格にする（理論の正本は [playbooks/effectuation.md](playbooks/effectuation.md)）。目的は事前に決めず、手段から行動して事後的に立ち上げる。予測（壁打ち）で確信を膨らませず、行動（試行・他者反応）で確信を育てる（証拠の非対称性）。

| やりたいこと | スキル | 原則／相 |
|---|---|---|
| 新しい個人案件を雛形から作成する | `/new-person` | セットアップ |
| 漠然としたキャリア観から目的仮説の種を引き出す | `/surface` | Reason（種出し） |
| 種を反証可能な目的仮説に精錬する（1問ずつ） | `/formulate-purpose` | Reason |
| 手中の鳥（制約C）を数え、目的に接地する | `/hand` | ①手中の鳥 |
| 許容可能な損失で次の試行を設計する | `/afford` | ②許容可能な損失 |
| パートナーを洗い出し事前コミットを取りにいく打診を設計する | `/quilt` | ③クレイジーキルト |
| 生データ（壁打ち／試行／第三者反応）を取り込み内省し、外界の証拠なら確信度を更新 | `/reflect` | Act実施→Observe |
| 驚き・想定外・偶然を逆手に取り、新しい目的の種・ピボットに変える | `/lemonade` | ④レモネード／explore |
| 予測・過信への逃避を突き、根拠なき確信度を引き下げ、行動へ引き戻す | `/pilot` | ⑤飛行機のパイロット／撹乱 |
| ピボット・巻き戻し・棚上げ・意図的引き下げ・モード切替を記録 | `/decide` | 岐路 |
| 系譜・試行ボード・関係グラフを可視化 | `/view` | 俯瞰 |
| Wikiの健全性チェック（証拠非対称ルール含む） | `/lint` | 保守 |
